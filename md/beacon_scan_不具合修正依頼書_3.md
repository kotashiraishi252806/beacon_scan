## 2. 不具合①：複数AP選択時、接続許可ダイアログの候補が1件に絞られてしまう

### 再現手順

1. アプリでスキャンを実行する（手動スキャン）
2. 検出結果一覧で「全て選択」ボタンを押す
3. 「選択したAPを測定」ボタンを押す

### 期待する動作

選択した全AP（複数）に対して、順番に接続測定（Supplicant状態測定）が行われる。

### 実際の動作

SSIDリストの1番上のSSIDから接続測定が始まるが、Androidシステムの制約により、ユーザーに接続許可を求めるダイアログのSSID候補が、そのSSID1件のみに絞り込まれてしまう。

### 推定原因（要調査）

`MainActivity.kt` の `measureSupplicantForAp()`（873〜976行目）が、選択されたAP1台ごとに個別のNetworkRequestを作成し、`connectivityManager.requestNetwork()` を呼び出す実装になっています。

```kotlin
// buildSupplicantSpecifier(ap) 978〜1002行目
WifiNetworkSpecifier.Builder().setBssid(mac).setSsid(ssid) ...
```

BSSID・SSIDを1つに指定したNetworkSpecifierでrequestNetwork()を呼ぶと、Android OS側の仕様上、システムの接続確認ダイアログにはその条件に一致するネットワークのみが表示されます。そのため実装としては「AP1台ずつ・都度システムダイアログでユーザー許可を得る」形になっており、「複数APを一括選択して連続測定する」というアプリ側のUI上の期待と、OSの制約との間にギャップが生じていると考えられます。

### 依頼事項

- この制約がAndroid OS仕様として回避不可能なものか、実装変更（NetworkSpecifierの指定方法、WifiNetworkSuggestion APIの利用など）で軽減可能かを調査してください。
- 回避できない場合は、「AP毎に接続確認ダイアログが表示される」旨を測定開始前にユーザーへ明示するなど、UX面での改善（案内文言・進捗表示の見直し等）を検討してください。
- この現象が発生している最中に、不具合②（中断ボタン消失）が併発していないか、合わせて調査してください。

## 3. 不具合②：測定中に「測定中断」ボタンが消え、中断操作ができなくなる

### 再現手順

不具合①と同じ手順（全て選択→選択したAPを測定）で測定を開始し、Android接続確認ダイアログの表示中〜複数AP連続測定中における「測定中断」ボタンの状態を確認する。

### 期待する動作

測定処理中は常に「測定中断」ボタンが画面に表示され、いつでもタップして測定を中断できる。

### 実際の動作

測定中に「測定中断」ボタンが画面から消え、中断操作ができなくなる。

### 推定原因（要調査）

`btnStopMeasurement` の表示制御は以下の2か所のみで行われています。

```kotlin
// btnMeasureSelected.setOnClickListener 内（425〜427行目付近）
btnStopMeasurement.visibility = View.VISIBLE  // 測定開始時

// lifecycleScope.launch { ... } 内（454行目付近）
btnStopMeasurement.visibility = View.GONE     // 全AP測定完了後
```

コード上、測定処理の途中で明示的にGONEへ変更している箇所は見当たりません。そのため、以下のようなActivityのライフサイクルに起因する現象が疑われます。

- Android接続確認システムダイアログの表示に伴いActivityがonStop / onDestroyされる場合があり、システムによってActivityが再生成されると、`btnStopMeasurement` は `activity_main.xml` 上の初期値（`android:visibility="gone"`）に戻ってしまう。
- 測定処理は `lifecycleScope.launch`（Activityのライフサイクルに紐づくコルーチンスコープ）で起動されているため、Activityが再生成されると、実行中の測定処理とUI（再生成後の新しいボタンインスタンス）との紐付けが失われる可能性がある。
- 画面回転やバックグラウンド遷移など、その他のライフサイクルイベントによるActivity再生成も同様の要因になり得る。

### 依頼事項

- Android接続確認ダイアログの表示時に、Activityのライフサイクルイベント（onPause / onStop / onDestroy）がどのタイミングで発火するか実機で確認してください。
- Activityの再生成が原因である場合、測定処理をActivityのライフサイクルに依存しない形（例：ViewModel + viewModelScopeへの移行、または長時間処理としてForeground Service化）に変更するか、状態保持（onSaveInstanceState等）により再生成後もボタン表示・処理状態を正しく復元できるよう修正してください。
- 修正後は、画面回転や一時的なバックグラウンド遷移を挟んでも「測定中断」ボタンが常時表示・操作可能であることをテストしてください。

## 4. 追加要望①：選択方法によらず、測定中の並び替え動作が同じであることの確認

### 内容

「選択したAPを測定」ボタンでの測定時、測定中のAPが一覧の一番上に来て、測定完了後に一番下へ移動する動き（`onApStart` / `onApFinished`、MainActivity.kt 437〜451行目）が、「全て選択」ボタンで選んだ場合だけでなく、チェックボックスを個別にタップして選んだ場合でも同じように起きることを確認してください。

### 補足

コードを確認した限り、選択方法（「全て選択」ボタン／個別チェックボックス）に関わらず、最終的に `adapter.getSelectedItems()`（選択中のbssid集合）を対象に同じ処理が走るため、実装上は既に選択方法によらず同一の動作になっているはずです。実機での動作確認のみで問題ないと想定していますが、もし選択方法によって挙動差がある場合はご報告ください。

## 5. 追加要望②：測定状態に応じたリスト項目の色分け表示

### 内容

「選択したAPを測定」ボタンによる**手動選択フロー**において、一覧の各AP項目（bssid・ssid・rssiなどが表示されている1件分の行＝リストアイテム）を以下のように色分けし、未測定・測定中・測定済みを視覚的に区別できるようにしてください。

※「手動選択フロー」とは、「全て選択」ボタンで一括選択した場合・チェックボックスで個別に選択した場合の**両方**を含みます（測定自体はどちらも同じ「選択したAPを測定」ボタンから開始するため）。自動スキャン（switchAutoScan ON）のときは今回の色分けの対象外とし、現状の表示のままとしてください。

| 状態 | 表示 |
|---|---|
| 未測定（まだ測定順が回ってきていない） | 現状の見た目のまま（変更なし） |
| 測定中（現在接続測定を行っている1件） | 現状の見た目のまま（変更なし） |
| 測定済み（測定が完了した項目） | 暗めのグレーを50%透過で重ねた表示にする |

補足：測定中は選択モードが解除されチェックボックスが非表示になる（`btnMeasureSelected.setOnClickListener` 内、415〜418行目）ため、測定開始後はこの色分けが進捗状況を伝える唯一の手がかりになります。

### 実装案（参考）

`AccessPoint.supplicantFinalState` は初期値が `"NOT_MEASURED"` で、測定完了時（`measureSupplicantForAp()` 内）に別の値へ更新されます。この値を使って、`ApAdapter.onBindViewHolder()`（MainActivity.kt 1327行目〜）で以下のような判定を行うことで実現できると考えられます。

```kotlin
if (ap.supplicantFinalState != "NOT_MEASURED") {
    // 測定済み：暗いグレーを50%透過で重ねる
} else {
    // 未測定・測定中：現状のまま
}
```

「測定中」の1件だけを別扱いする必要はなく、`supplicantFinalState` が更新されるのは測定完了後（`onApFinished` のタイミング）のみのため、上記の判定だけで「未測定・測定中は現状の色／測定済みのみグレー」という要望を満たせるはずです。実装しやすい方法であれば上記に限らず問題ありません。

### 適用範囲

自動スキャン（switchAutoScan ON）時は対象外です。現状「並び替えなし」の実装のままで問題ありません。今回の色分けは手動選択フローのみに適用してください。

## 6. 実装後の検証で見つかった追加不具合（チェックボックスで個別選択して測定した場合）

上記「5. 追加要望②」の実装後、チェックボックスで一部のAPを個別に選択して測定する動作を検証したところ、以下2点の不具合が見つかりました。

### 不具合③：測定が完了したAP項目（リストアイテム）の色が灰色にならない

- 再現手順: チェックボックスで一部のAPを選択 →「選択したAPを測定」で測定を実行
- 期待する動作: 測定が完了したAP項目（bssid・ssid・rssiなどが表示されている行）が、暗いグレー50%透過の表示になる（本書5.の内容）
- 実際の動作: 測定完了後もAP項目の色が変わらない
- 依頼事項: 「5. 追加要望②」で依頼した色分けが実装されているか、また `supplicantFinalState` の更新タイミングと `onBindViewHolder()` での再描画（`notifyDataSetChanged()` 等）が正しく連動しているかを確認し、修正してください。

### 不具合④：選択していないAPも測定中の一覧に表示され、選択したAPがどれかわかりにくい

- 再現手順: 同上
- 期待する動作: 測定中の一覧には、選択して測定対象になっているAPが分かりやすく表示される
- 実際の動作: 選択していないAPも一覧に混在して表示されるため、どれが測定対象として選ばれたAPなのか判別しにくい
- 推定原因（要調査）: `MainActivity.kt` の `btnMeasureSelected.setOnClickListener`（408〜477行目）では、測定対象は `selectedAps = adapter.getSelectedItems()` に絞り込まれていますが、画面表示に使っている `apList` 自体は、スキャンで検出した全AP（選択有無に関わらず）を保持したままになっています。そのため測定中は「全AP表示＋一部だけ色付け」という形になり、選択したAPだけを判別しづらくなっていると考えられます。また `tvSsidCount` の「今回検出: n件」表示も、選択件数ではなく全体のスキャン件数（`apList.size`）を表示しているため、同様に紛らわしい原因になっている可能性があります。
- 依頼事項:
  - 測定開始時に、一覧の表示を「選択されたAPのみ」に絞り込むこと（`apList` を選択済みAPのみに置き換える、または測定専用の別リストで表示するなど）を検討してください。
  - `tvSsidCount` などの件数表示も、全体のスキャン件数ではなく、選択して測定するAP数を反映するよう見直してください。
  - 上記の対応が難しい場合は、少なくとも「選択されたAPかどうか」が一覧内で明確に判別できる表示にしてください。
